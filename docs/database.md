# Database Layer: Prisma + ClickHouse + Migrations

The database layer (`packages/db`) is the foundation of OpenPanel's data persistence. It manages a dual-database architecture: **PostgreSQL via Prisma** for application metadata, and **ClickHouse** for high-volume analytics events. This document covers the schema, query patterns, migrations, and how to extend the layer.

## Key Files

| File | Purpose |
|------|---------|
| `packages/db/prisma/schema.prisma` | PostgreSQL schema (organizations, projects, users, reports, dashboards, cohorts, etc.) |
| `packages/db/src/clickhouse/client.ts` | ClickHouse client initialization, TABLE_NAMES constants, utility functions (`chQuery`, `chQueryWithMeta`) |
| `packages/db/src/clickhouse/query-builder.ts` | Custom `Query` class + `clix` helper for building type-safe ClickHouse queries (745 lines) |
| `packages/db/src/clickhouse/migration.ts` | ClickHouse table/view creation, clustering/replication logic for distributed setups |
| `packages/db/code-migrations/*.ts` | Numbered ClickHouse code migrations (1-18, idempotent, run via `pnpm migrate:deploy:code`) |
| `packages/db/prisma/migrations/` | Prisma SQL migrations (auto-generated by `pnpm migrate dev`) |
| `packages/db/src/services/*.ts` | Business logic repositories (chart, event, profile, session, cohort, funnel, etc.) |
| `packages/db/index.ts` | Public API: exports all services, query builder, Prisma client, ClickHouse client |

## Overview: Postgres vs ClickHouse

### PostgreSQL (via Prisma)
**Stores:** Application metadata and configuration
- Organizations, projects, users, members, access control
- Dashboards, reports, report layouts
- Clients (API keys), salts, integrations
- Event metadata (EventMeta), custom event definitions (CustomEvent)
- Cohorts definitions (logical rules, stored as JSON)
- Notification rules, shares, invites
- Imports tracking (job status, progress)
- Materialized column metadata (for ClickHouse optimization)
- Code migration history (__code_migrations table)

**Why:** Relational queries, transactions, soft-deletes, schema flexibility, subscription tracking.

### ClickHouse
**Stores:** Analytics events and profiles (high cardinality, time-series)
- **events** (ReplicatedMergeTree): Raw event stream (~1B+ rows per project)
- **profiles** (ReplacingMergeTree): User profile snapshots with properties
- **sessions** (ReplacingMergeTree): Session-level aggregations
- **cohort_members** (ReplacingMergeTree): Cohort membership (profile_id → cohort_id)
- **events_imports_v2** (MergeTree): Imported events from external sources
- Multiple materialized views (MVs) for denormalized aggregations:
  - `profile_event_summary_mv`: Per-profile event counts
  - `distinct_event_names_mv`: Distinct event name list
  - `event_property_values_mv`: Property value cardinality
  - `dau_mv`: Daily active users
  - `session_replay_chunks`: Session replay storage (ClickHouse native)

**Why:** Sub-second OLAP queries on billions of rows, compression, partitioning, columnar storage.

## Prisma Schema Deep Dive

The Prisma schema defines 26 models covering user accounts, organizations, projects, and analytics metadata.

### Core Models

#### User, Account, Member
```prisma
model User {
  id: String @id
  email: String @unique
  firstName/lastName: String?
  createdOrganizations: Organization[] (relation)
  membership: Member[]
  accounts: Account[] (OAuth providers)
  sessions: Session[]
}
```
**Key:** Single sign-on support via `Account` (OAuth provider, tokens), sessions for auth tokens.

#### Organization & ProjectAccess
```prisma
model Organization {
  id: String @id
  name: String
  projects: Project[]
  members: Member[]
  // Subscription fields (Stripe integration)
  subscriptionId, subscriptionStatus, subscriptionEndsAt, etc.
  // Deletion scheduling
  deleteAt: DateTime?
}

model ProjectAccess {
  projectId, organizationId, userId: String
  level: AccessLevel (read | write | admin)
}
```
**Key:** Org contains projects. Users access projects via ProjectAccess at a specific level. Subscription tracking in org.

#### Project
```prisma
model Project {
  id: String @id
  name: String
  organizationId: String
  eventsCount: Int (cached)
  types: ProjectType[] (website | app | backend)
  domain: String? cors: String[]
  filters: Json (IPrismaProjectFilters)
  clients: Client[]
  reports: Report[]
  dashboards: Dashboard[]
  meta: EventMeta[] (per-project event metadata)
}
```
**Key:** Projects contain events, dashboards, and reports. EventMeta stores event names/icons per project.

#### Dashboard & Report
```prisma
model Dashboard {
  id, name, projectId: String
  reports: Report[]
}

model Report {
  id, name, projectId: String
  interval: Interval (hour | day | month | minute | week)
  chartType: ChartType (linear | bar | histogram | pie | metric | funnel | retention)
  events: Json, breakdowns: Json
  formula: String?, unit: String?
  metric: Metric (sum | average | min | max)
  globalFilters, holdProperties, hiddenSeries: Json
  ttcAggregation: String ("avg" default)
  layout: ReportLayout? (grid position)
}
```
**Key:** Reports define chart queries. Engine translates these to ClickHouse queries via `executeChart()`.

#### EventMeta & CustomEvent
```prisma
model EventMeta {
  id: String @id
  name: String (unique per project)
  conversion: Boolean? (marks as conversion event)
  color, icon: String?
  projectId: String
}

model CustomEvent {
  id: String @id
  name: String (unique per project)
  projectId: String
  definition: Json (IPrismaCustomEventDefinition)
  conversion: Boolean
}
```
**Key:** `EventMeta` is UI metadata (icons). `CustomEvent` defines compound event rules (e.g., "purchase > $100").

#### Cohort
```prisma
model Cohort {
  id: String @id
  name, description: String?
  projectId: String
  definition: Json (CohortDefinition)
  isStatic: Boolean
  profileCount: Int
  lastComputedAt: DateTime?
  computeOnDemand: Boolean
}
```
**Key:** Logical definition (JSON, e.g., "users who viewed page X AND purchased > once"). Membership materialized to ClickHouse `cohort_members` table.

#### Imports
```prisma
model Import {
  id: String @id
  projectId: String
  jobId: String? (BullMQ job ID)
  status: ImportStatus (pending | processing | completed | failed)
  config: Json (import source config)
  totalEvents, processedEvents: Int
  currentStep, currentBatch: String?
}
```
**Key:** Tracks data import jobs. Events land in ClickHouse `events_imports_v2` table.

#### MaterializedColumn
```prisma
model MaterializedColumn {
  targetTable: String ("events" | "profiles")
  propertyKey: String (e.g., "utm_source")
  columnName: String
  cardinality: Int
  usageCount: Int
  benefitScore: Float
  status: String ("active" | "pending" | "failed")
  materializedAt: DateTime?
}
```
**Key:** Optimization metadata. Service `MaterializeColumnsService` analyzes usage and materializes high-cardinality properties as real columns in ClickHouse.

See `packages/db/prisma/schema.prisma:1-570` for full schema.

## ClickHouse: Tables and Views

### events (Primary Event Stream)
```sql
CREATE TABLE events (
  id UUID,
  name LowCardinality(String),
  sdk_name, sdk_version LowCardinality(String),
  device_id, profile_id, project_id, session_id String (CODEC ZSTD/LZ4),
  path, origin, referrer String,
  referrer_type LowCardinality(String),
  duration UInt64 (Delta+LZ4),
  properties Map(String, String) (CODEC ZSTD),
  created_at DateTime64(3),
  country LowCardinality(FixedString(2)),
  city, region String,
  ... more columns ...
)
ENGINE = ReplicatedMergeTree (in clustered) / MergeTree (in self-hosted)
PARTITION BY toYYYYMM(created_at)
ORDER BY (project_id, created_at, profile_id)
```
**Key:** Time-partitioned, ordered by project_id for fast filtering. Properties stored as Map(String, String) for flexible schema.

### profiles (User Snapshots)
Stores latest profile state with properties (email, name, custom fields, etc.) using ReplacingMergeTree for efficient updates.

### sessions (Session-Level Aggregations)
Stores start, end, event count, duration per session.

### cohort_members
Stores profile membership: `(project_id, cohort_id, profile_id, matched_at, version)`. Used to filter profiles by cohort.

### events_imports
Similar to events, stores imported events with metadata flag to distinguish historical imports.

### Materialized Views
Built incrementally from events table via CTEs:
- **profile_event_summary_mv**: `(project_id, profile_id, event_name, count, last_seen_at)` — fast cohort filtering by event frequency
- **distinct_event_names_mv**: Distinct `(project_id, event_name)` — for UI dropdowns
- **event_property_values_mv**: `(project_id, property_key, value_count)` — cardinality estimation

See `packages/db/code-migrations/4-cohorts.ts` and `3-init-ch.ts` for full schema definitions.

## Query Builder: `clix` (Fluent ClickHouse Query DSL)

All ClickHouse queries **must** use the custom query builder at `packages/db/src/clickhouse/query-builder.ts` (per `.cursorrules`).

### Why Custom Builder?
1. **Type safety:** Compile-time validation of SQL structure
2. **SQL injection protection:** Automatic parameter escaping via `sqlstring`
3. **ClickHouse-specific optimizations:** Timezone handling, date functions, CTEs, distributed queries
4. **Readability:** Fluent API vs string concatenation

### Basic Usage

```typescript
import { ch, clix } from '@openpanel/db';

const query = clix(ch, 'America/New_York')
  .select(['project_id', 'count() as cnt'])
  .from('events')
  .where('project_id', '=', projectId)
  .where('created_at', '>=', startDate)
  .groupBy('project_id')
  .orderBy('cnt', 'DESC')
  .limit(100);

const results = await query.execute();
```

### Core Methods

| Method | Returns | Example |
|--------|---------|---------|
| `select(columns)` | Query | `.select(['id', 'count()'])` |
| `from(table)` | Query | `.from('events')` |
| `where(col, op, val)` | Query | `.where('project_id', '=', id)` |
| `andWhere(col, op, val)` | Query | `.andWhere('name', 'LIKE', '%foo%')` |
| `orWhere(col, op, val)` | Query | `.orWhere('status', '=', 'active')` |
| `whereGroup()` | WhereGroupBuilder | `.whereGroup().where(...).orWhere(...).end()` — returns a WhereGroupBuilder; chain conditions then call `.end()` to return to the Query (use `orWhereGroup()` for an OR-joined group) |
| `groupBy(cols)` | Query | `.groupBy(['project_id', 'toDate(created_at)'])` |
| `having(col, op, val)` | Query | `.having('count()', '>', 10)` |
| `orderBy(col, dir)` | Query | `.orderBy('count()', 'DESC')` |
| `limit(n)` | Query | `.limit(100)` |
| `offset(n)` | Query | `.offset(50)` |
| `join(table, condition, alias?)` | Query | `.join('profiles', 'profiles.id = events.profile_id')` — INNER join; type is the method name |
| `leftJoin(table, condition, alias?)` | Query | `.leftJoin('profiles', 'profiles.id = events.profile_id')` (also `innerJoin`, `rightJoin`, `fullJoin`, `crossJoin`, `leftAnyJoin`) |
| `with(name, query)` | Query | `.with('recent', recentEventsQuery)` — add common table expression |
| `union(otherQuery)` | Query | `.union(otherQuery)` |
| `from(table, final?)` | Query | `.from('profiles', true)` — pass `true` for FINAL (ReplacingMergeTree deduplicated read) |
| `toSQL()` | string | Render query string |
| `execute<T>()` | Promise<T[]> | Execute and fetch results |

### Date Helpers (Static Methods on `clix`)

```typescript
clix.date('2025-01-15')              // '2025-01-15' (Date only)
clix.datetime('2025-01-15T10:30:00') // '2025-01-15 10:30:00' (DateTime)
clix.toStartOf('created_at', 'day')  // toStartOfDay(created_at)
clix.toStartOfInterval('created_at', 'day', origin)  // toStartOfInterval(...) for custom bucketing
clix.dynamicDatetime(date, 'month')  // toDate() for month/week, toDateTime() for hour/minute
```

### Expression Wrapping

For raw SQL expressions:
```typescript
const expr = clix.exp('COUNT(DISTINCT profile_id)');
query.select([expr as any]);

// Or with parentheses:
query.where('duration', '>', clix.exp('avg(duration)'));
```

### Example: Event Funnel Query

```typescript
const funnel = clix(ch, timezone)
  .select([
    `toDate(created_at) as date`,
    `event_name`,
    `count(DISTINCT profile_id) as unique_users`
  ])
  .from('events')
  .where('project_id', '=', projectId)
  .where('created_at', '>=', startDate)
  .where('created_at', '<', endDate)
  .where('event_name', 'IN', ['page_view', 'add_to_cart', 'purchase'])
  .groupBy(['date', 'event_name'])
  .orderBy('date', 'ASC')
  .orderBy('event_name', 'ASC');

const results = await funnel.execute<{ date: string; event_name: string; unique_users: number }>();
```

See `packages/db/src/clickhouse/query-builder.ts:1-745` for full Query class definition.

## Migrations: Two Systems

OpenPanel has two independent migration systems:

### 1. Prisma Migrations (PostgreSQL)
**Location:** `packages/db/prisma/migrations/`  
**Generated:** Auto-created by `pnpm migrate dev` when you edit `schema.prisma`  
**Run:** `pnpm migrate` (development), `pnpm migrate:deploy:db` (production)

Example migration (file):
```bash
20240105_add_histogram/migration.sql
```

**Process:**
1. Edit `prisma/schema.prisma`
2. Run `pnpm migrate dev --name add_histogram`
3. New file generated in `prisma/migrations/[timestamp]_add_histogram/migration.sql`
4. In CI/deployment: `pnpm migrate:deploy:db` runs all unapplied migrations

**Key:** Idempotent, applied in order, persisted in `_prisma_migrations` table.

### 2. Code Migrations (ClickHouse)
**Location:** `packages/db/code-migrations/*.ts`  
**Format:** Numbered files (`1-settings.ts`, `2-accounts.ts`, ... `18-move-first-event-to-events.ts`)  
**Generated:** Written by hand — define ClickHouse DDL and data transformations  
**Run:** `pnpm migrate:deploy:code` (or `pnpm migrate:deploy:code --cluster` for clustered CH)

Example migration:
```typescript
// 4-cohorts.ts
export async function up() {
  const isClustered = getIsCluster();
  const sqls: string[] = [];
  
  sqls.push(
    ...createTable({
      name: 'cohort_members',
      columns: ['project_id String', ...],
      engine: 'ReplacingMergeTree(version)',
      distributionHash: 'cityHash64(project_id, cohort_id)',
      replicatedVersion: '1',
      isClustered,
    })
  );
  
  await runClickhouseMigrationCommands(sqls);
  // No need to save — migrate.ts auto-records in __code_migrations
}
```

**Process:**
1. Create `packages/db/code-migrations/[N]-name.ts` with `export async function up()`
2. Use `createTable()`, `createMaterializedView()`, `addColumns()`, etc. from `migration.ts`
3. Run `pnpm migrate:deploy:code` (auto-detects new migrations, runs only unapplied)
4. Recorded in PostgreSQL `__code_migrations` table (Prisma model: `CodeMigration`)

**Key:** Idempotent (check `IF NOT EXISTS`), handles both clustered and self-hosted CH, data transformations in code.

**Flags:**
- `--cluster`: Enable clustered mode (replicated tables, distributed queries)
- `--dry`: Print migration SQL but don't run
- `--no-record`: Don't record migration in `__code_migrations` (for testing)

See `packages/db/code-migrations/migrate.ts:17-111` for orchestration logic.

## Chart Engine: The Pipeline

The chart engine (`packages/db/src/engine/index.ts`) orchestrates the query flow:

1. **normalize:** Transform user input (IChartInput) into a normalized plan
2. **plan:** Build execution steps (which events, breakdowns, filters, CTEs)
3. **fetch:** Execute ClickHouse queries for current period (calls `clix`)
4. **compute:** Evaluate formulas (e.g., "Series1 / Series2")
5. **format:** Return FinalChart with metadata

```typescript
const result = await executeChart({
  projectId: 'proj-123',
  events: [{ name: 'purchase' }],
  breakdowns: [{ name: 'utm_source' }],
  startDate: new Date('2025-01-01'),
  endDate: new Date('2025-01-31'),
  interval: 'day',
  previous: true, // Include previous period
});
```

## How It Connects

### API Layer (apps/api)
- Receives HTTP requests → calls services from `@openpanel/db`
- Example: `POST /api/chart` calls `ChartEngine.execute()` 
- `AuthService` uses `db.user.findUnique()` for auth checks

### Worker (apps/worker)
- Polls jobs from Redis queue (groupmq + BullMQ)
- Import jobs: calls `ImportService.createImport()`, inserts to `events_imports`, updates progress
- Session replay: inserts replay chunks to ClickHouse
- Materialization: runs `MaterializeColumnsService` to analyze and ALTER TABLE

### Dashboard (apps/public)
- Sends chart requests to API (which calls ChartEngine)
- Stores user dashboards/reports in Prisma

### Admin Tools (tooling/)
- Direct database access for analytics, cleanup

## Gotchas

1. **Query Builder vs Raw SQL:**
   - `.cursorrules` requires `clix` for ClickHouse queries, not raw strings
   - Exception: When `chQuery()` is more readable and no builder method exists
   - Always use `sqlstring.escape()` for raw SQL values

2. **Clustered vs Self-Hosted Tables:**
   - Clustered: Tables created as `*_replicated` (ReplicatedMergeTree) + `*` (Distributed)
   - Self-hosted: Single table, no replication
   - Migrations check `getIsCluster()` and generate correct DDL
   - Always use `getReplicatedTableName(table)` when mutating (INSERT/ALTER)

3. **Materialized Views are Incremental:**
   - MVs populate `slowly` from new events to avoid OOM
   - Pass `from(table, true)` (FINAL) when querying ReplacingMergeTree to deduplicate
   - Don't assume MV is up-to-date for recent data

4. **Cohort Membership:**
   - Stored in ClickHouse `cohort_members` table, not computed on-the-fly
   - Run `CohortService.computeCohortMembership()` after cohort definition changes
   - `computeOnDemand` flag in Prisma Cohort model skips materialization (only for small cohorts)

5. **Properties are Flexible (Map):**
   - Event `properties` is `Map(String, String)` in ClickHouse
   - No schema validation at insert time
   - Materialized columns optimize high-cardinality properties

6. **Timezone Handling:**
   - Query builder accepts timezone in constructor: `clix(ch, 'America/New_York')`
   - User timezone stored in Organization model
   - Pass it through the chart engine for correct bucketing

7. **Subscription Limits:**
   - Org model tracks `subscriptionPeriodEventsCount` and `subscriptionPeriodEventsLimit`
   - Chart engine limits results if subscription limit exceeded (see `getOrganizationSubscriptionChartEndDate`)

8. **Prisma Extensions:**
   - Custom computed fields on Organization (e.g., `isActive`, `isTrial`, `subscriptionCurrentPeriodStart`)
   - Don't query raw; use these computed fields for business logic
   - Read replica extension allows read-only queries to scale
