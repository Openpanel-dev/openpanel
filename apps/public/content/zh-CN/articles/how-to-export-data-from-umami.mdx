使用 [web analytics](/features/web-analytics) 时，你希望对自己的数据拥有控制权。如果你要切换平台、创建备份，或者在其他工具中分析数据，就需要一种方法将数据从 Umami Analytics 中导出。

Umami 的数据导出方式会因你使用的是云服务还是自托管而不同。本指南将分别介绍这两种情况，并解释你为何可以考虑 OpenPanel 这样的替代方案。

## 了解 Umami 的导出限制

Umami Analytics 是一款不错的、注重隐私的分析工具，但在数据可移植性方面存在限制。云版本包含导出功能，如果你正在为其服务付费，这非常方便。不过，如果你自托管 Umami，则没有官方导出工具。

之所以存在这一差距，是因为 Umami 团队更专注于分析功能，而不是数据迁移工具。可以理解，但对于需要迁移数据的用户来说，这确实是个问题。

许多自托管用户只有在需要迁移或创建备份时，才会发现这一限制。你可能已经把数月甚至数年的宝贵分析数据锁在 PostgreSQL 数据库中，却没有简单直接的方法将其导出。

## 云端导出功能

如果你使用的是 Umami Cloud，就可以使用其内置导出功能。流程很简单，但有一些注意事项。

要从 Umami Cloud 导出，先登录你的账户，然后进入 Account 部分。在那里你会看到一个 Data 标签页，其中包含导出选项。

提交导出请求后，Umami 会在后台处理，因此你不必在数据准备期间一直等待。导出准备好后，他们会通过电子邮件向你发送下载链接。

导出的数据采用 CSV 格式，并使用 gzip 压缩以减小文件大小。如果你的网站流量很大，拥有数百万次页面浏览，这会很有帮助。压缩可将文件大小减少 70-90%，让下载更快。

不过也有一些限制。每个导出请求一次只能处理一个网站。如果你跟踪多个域名，就需要为每个域名单独提交导出请求；如果你管理很多网站，这会很麻烦。

大型数据集的处理时间也可能很长。Umami 不提供实时进度更新，所以你只能等待邮件通知。对于非常大的站点，导出可能需要数小时才能完成。

## 自托管方案：OpenPanel 导出器

对于自托管 Umami 用户，我们开发了 [@openpanel/umami-exporter](https://www.npmjs.com/package/@openpanel/umami-exporter) 来填补这一空缺。它可直接连接到你的 PostgreSQL 数据库，并将所有分析数据提取为 CSV 格式。

你无需永久安装任何东西，也无需修改 Umami 安装。该工具独立运行，只需要对数据库的只读访问权限。

使用 npx，你可以无需安装即可运行导出器：

```bash
npx op-umami-exporter "postgresql://username:password@host:port/database"
```

这一条命令就能完成所有事情：连接数据库、查询所需表、关联数据，并输出可直接导入其他系统的格式。

导出器会从多个表中提取数据，包括 sessions、events 和 website configurations，然后将它们组合成非规范化格式。这让导出的数据更便于在其他工具中使用。

一个常见障碍是，Umami 的默认 Docker 配置没有暴露 PostgreSQL 端口。你需要修改 docker-compose.yml 文件，添加端口映射。这是一次性更改，只需一分钟：

```yaml
services:
  postgres:
    ports:
      - "5432:5432"
```

完成此更改并重启容器后，导出器就可以连接到你的数据库了。默认的 Umami 凭据通常是用户名和密码都为 umami，不过你应该在配置中确认这一点。

## 为什么考虑从 Umami 迁移？

Umami 在注重隐私的网站分析方面做得不错，但你可能需要更高级的功能。这就是 OpenPanel 这类平台发挥作用的地方。

我们为需要超越页面浏览跟踪的网站构建了 OpenPanel。它保留了与 Umami 相同的隐私优先理念，并增加了更深入了解用户行为的功能。

其中一个区别是实时数据处理。你能看到访客活动正在发生，而不是像批处理系统那样存在延迟。这有助于你快速响应流量激增、营销活动或技术问题。

我们的 [event tracking](/features/event-tracking) 不仅限于页面浏览。你可以跟踪自定义事件、用户交互以及 [conversion funnels](/features/funnels)，而无需编写复杂代码。OpenPanel 会自动捕获许多在 Umami 中需要手动设置的交互。

筛选和细分也更高级。你可以为特定用户群体构建查询，更灵活地比较时间段，并创建专注于你关心指标的自定义仪表盘。

对于团队来说，OpenPanel 具备更好的协作功能。你可以共享特定视图，为利益相关者创建只读访问权限，并设置自动报告，让所有人都能及时了解情况，而无需授予完整的分析权限。

## 做出迁移决定

是否从 Umami 迁移，取决于你的分析需求以及你预期的增长方式。

如果你运行的是一个简单的博客或作品集网站，Umami 的基础分析可能就足够了。它的简洁性让你能专注于内容创作。

但如果你运营的是电商网站、SaaS 应用，或者任何用户行为会直接影响收入的业务，你很快就会遇到 Umami 的限制。无法跟踪自定义事件或构建转化漏斗会成为瓶颈。

同时也要考虑总体拥有成本。自托管 Umami 看起来是免费的，但你仍然要为服务器资源、维护时间以及错失洞察的代价买单。一旦把这些隐性成本算进去，像 OpenPanel 这样的云平台通常会更有价值。

上面的导出流程让迁移的风险更低。你可以导出历史数据，试用 OpenPanel 或其他平台，并根据实际体验而不是功能列表来做决定。

## 数据导出的最佳实践

导出 Umami 数据时，时机很重要。请选择低流量时段，以尽量减少对数据库性能的影响。导出过程是只读的，不应影响你的实时分析，但谨慎一些总是更好。

导出完成后，一定要立即验证结果。打开 CSV 文件并检查数据是否正确。查看日期范围是否准确，确认所有网站都已包含，并检查事件数据的格式是否正确。

对于大型数据集，可以考虑按日期范围分批导出。这会让文件更易管理，并降低超时问题的风险。如果需要，之后你可以轻松将多个 CSV 文件合并。

妥善保护你的导出数据。这些文件包含你网站访客的信息，虽然 Umami 不收集个人数据，但汇总信息仍可能具有敏感性。如果你要长期存储或跨网络传输，请对导出文件加密。

定期导出是很好的备份方式。即使你对 Umami 很满意，定期导出也能防止数据丢失。你可以设置每月提醒来导出数据，或者用一个简单脚本自动完成这一过程。
